DataMesh Logística

MICROSOFT FABRIC | DATA MESH | REAL-TIME ANALYTICS

01. El Desafío: Fragmentación de Datos

Una compañía logística con múltiples operadores (DHL, FedEx, UPS, LocalPost) enfrentaba un problema crítico: datos fragmentados en silos incompatibles. Cada dominio (Transacciones, Logística, Gobernanza) operaba de forma aislada, imposibilitando una visión unificada del rendimiento.

La pregunta estratégica era clara: ¿Cómo unificar tres dominios de datos con diferentes latencias (batch + near real-time) sin duplicar almacenamiento?

02. Arquitectura Data Mesh en Microsoft Fabric

Diseñé una arquitectura zero-copy utilizando OneLake Shortcuts para acceder a datos en tiempo real sin replicación. El resultado: un único Modelo Semántico que combina datos de lote y streaming.

┌─────────────────────────────────────────────────────────────┐ │ DATA MESH ARCHITECTURE │ ├─────────────────────────────────────────────────────────────┤ │ │ │ DOMAIN 1: TRANSACTIONS (D1) │ │ └─→ Lakehouse (Delta Lake) │ │ Sales records, revenue data (Batch) │ │ │ │ DOMAIN 2: LOGISTICS (D2) │ │ └─→ KQL Database (Eventhouse) │ │ Carrier status, delivery times (Near Real-Time) │ │ │ │ DOMAIN 3: GOVERNANCE (D3) │ │ └─→ KQL Database (Eventhouse) │ │ System logs, SLA compliance (Near Real-Time) │ │ │ │ CONSUMPTION LAYER (D4) │ │ └─→ Lakehouse with OneLake Shortcuts → D2, D3 │ │ Zero-copy data access, unified semantic model │ │ │ │ SEMANTIC MODEL │ │ └─→ Power BI Dataset with DAX measures │ │ Cross-domain relationships, KPIs │ │ │ └─────────────────────────────────────────────────────────────┘

ONE_LAKE_SHORTCUTS

Acceso zero-copy a datos en KQL Database. Los analytics reflejan siempre el dato operacional más reciente sin duplicación de almacenamiento.

KQL_DATABASE

Eventhouse para ingesta de alta velocidad. Maneja datos near real-time de logística y gobernanza con latencia sub-segundo.

DELTA_LAKE

Almacenamiento transaccional en Lakehouse. Datos de ventas y revenue en formato Delta para ACID compliance.

DAX_MEASURES

Medidas calculadas para KPIs de negocio: [Tiempo Promedio Entrega], [Tasa Incumbimiento SLA], [Total Envíos].

03. Integración Multi-Dominio

Tres dominios de negocio unificados en un único modelo semántico, cada uno con su propia latencia y tecnología óptima:

DOMINIO CONTENIDO LATENCIA TECNOLOGÍA
D1: Transactions Sales & Revenue Records Batch Lakehouse (Delta Lake)
D2: Logistics Carrier Status, Delivery Times Near Real-Time KQL Database
D3: Governance System Logs, SLA Compliance Near Real-Time KQL Database

Código de Ingesta KQL (Ejemplo)

// Ingesta de datos de logística en KQL Database .create table LogisticaRaw ( EnvioID: string, Carrier: string, FechaSalida: datetime, FechaEntrega: datetime, Estado: string, SLA_Cumplido: bool ) // Consulta de rendimiento por carrier LogisticaRaw | where FechaSalida >= ago(30d) | summarize TotalEnvios = count(), PromedioDias = avg(toint(FechaEntrega - FechaSalida)), TasaSLA = sumif(1, SLA_Cumplido) * 100.0 / count() | group by Carrier

04. Dashboard Interactivo

El dashboard final permite análisis cruzado entre dominios con filtros interactivos por carrier, rango de fechas y región de entrega.

Visualizaciones Principales:

  • Mapa de Ventas (D1 + D2): Estrategia de inventario por región de entrega, correlacionando ventas con tiempos de entrega.
  • Auditoría SLA (D1 + D3): Riesgo tecnológico: fallos de sistema vs métodos de pago. Identifica patrones de incumplimiento.
  • Rendimiento de Carriers (D2): Auditoría costo/eficiencia. Ranking de operadores logísticos por velocidad y cumplimiento.
  • Tendencia Histórica (D2): Eficiencia de entrega en el tiempo. Planning y forecast de capacidad logística.

Filtros Interactivos:

  • Carrier Selector: Filtra todo el análisis por operador logístico (DHL, FedEx, UPS, LocalPost)
  • Date Range: Ajusta ventana temporal para análisis de tendencias
  • Region Filter: Enfoca en zonas de entrega específicas

05. Medidas DAX Implementadas

El modelo semántico incluye medidas DAX optimizadas para el cálculo de KPIs críticos de negocio:

// Tiempo Promedio de Entrega (días) Tiempo Promedio Entrega = AVERAGE(Logistica[FechaEntrega] - Logistica[FechaSalida]) // Tasa de Incumplimiento SLA (%) Tasa Incumbimiento SLA = DIVIDE( CALCULATE( COUNTROWS(Logistica), Logistica[SLA_Cumplido] = FALSE() ), COUNTROWS(Logistica), 0 ) * 100 // Total de Envíos Total Envíos = COUNTROWS(Logistica) // Revenue Total (cross-domain) Revenue Total = SUM(Transacciones[Revenue])

08. Key Learnings

  • OneLake Shortcuts habilitan analytics en tiempo real zero-copy, eliminando la necesidad de ETL tradicional para consolidación.
  • KQL Database maneja ingesta de alta velocidad de forma eficiente, ideal para datos operacionales de logística.
  • Relaciones cross-domain desbloquean insights imposibles en arquitecturas siloed (ej: correlacionar fallos de sistema con pérdida de ventas).
  • DAX optimization es crítico en modelos híbridos batch/real-time para mantener performance de queries.
